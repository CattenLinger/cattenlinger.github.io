​用 tinc 有好一段时间了。我的服务大多数都使用 tinc 组成的内网来承载。一直以来我是直接固定 IP 使用，设置成「单地址」模式。因为网络内不止固定 IP 的服务器，对于临时进入内网的设备来说这种方式就十分麻烦了：每新增一个设别，需要翻查哪些 IP 可用，然后分别设置服务器端和客户端的「客户端配置文件」。

​tinc 除了路由模式单地址使用，还能使用其他模式，可在设置文档内查阅其模式类型：

```
Mode = router | switch | hub (router)
             This option selects the way packets are routed to other daemons.

             router  In this mode Subnet variables in the host configuration files will be used to form a routing table.  Only
                     unicast packets of routable protocols (IPv4 and IPv6) are supported in this mode.

                     This is the default mode, and unless you really know you need another mode, don't change it.

             switch  In this mode the MAC addresses of the packets on the VPN will be used to dynamically create a routing table
                     just like an Ethernet switch does.  Unicast, multicast and broadcast packets of every protocol that runs
                     over Ethernet are supported in this mode at the cost of frequent broadcast ARP requests and routing table
                     updates.

                     This mode is primarily useful if you want to bridge Ethernet segments.

             hub     This mode is almost the same as the switch mode, but instead every packet will be broadcast to the other
                     daemons while no routing table is managed.
```

​switch 和 hub 都比较类似，一般建议使用 switch。switch 模式下 tinc 相当于一个交换机，可以有交换机功能；而 hub 模式仅仅只是一个 hub ，就像一个对水流什么都不做的管道，同时把所有包都广播到所有节点让各个节点自己看着办：对各个节点而言，内网流量一览无遗，但同时增加很多流量消耗，对一般组网使用来说没什么必要。使用交换机模式，可以传递二、三层流量，可实现 dhcp 、广播等功能，相当于一个普通路由器，对比起来对小型私人内网来说，实用性比较高。

接下来开始介绍一下如何使用 switch 模式搭配 dhcpd 来搭建一个 tinc 内网。

```
              │     ┌──────────┐          │          ┌──────────┐
		          ├────>│ Client 2 │					├─────────>│ Client 4 │

 							.                           .
```


## 准备

需要一个可以直接通过 IP 访问的服务器，其他 daemon 可以都在 NAT 后面。

首先所有的机器都需要安装 tincd ，文中将以 ubuntu 18.04 为例。

```
sudo apt install tinc
```

同时确认一下 `ip` 命令可用，这里我主要以 `ip` 命令作讲解。

安装完成后，开始配置服务器端。关于 dhcp 的部分后面再补充，这里先把网配好 （´・ω・｀）

## 服务端配置

```
sudo su 								#为了方便，我直接使用 root 身份做所有的操作。
cd /etc/tinc/netname/		# netname 是 tinc 网络实例的名称，请替换成自己喜欢的名字
nano tinc.conf
```

在 `tinc.conf` 内配置如下内容：

```
Name = NETNAME         # 给此节点起个名字
AddressFamily = ipv4   # 使用 ipv4 
Mode = switch          # 使用交换机模式
#Device = /dev/net/tun # 可选，明确使用 tun/tap 设备
```

需要注意是否有 `/dev/net/tun` 或者 [tun/tap](https://en.wikipedia.org/wiki/TUN/TAP) 设备支持，因为二层网络交换需要用到 tun/tap 设备。

接下来需要配置 tinc 的 daemon 启动脚本。tinc 在启动和停止的时候都会分别执行 `tinc-up` 和 `tinc-down` 来做相对应的网络配置。

编辑 `tinc-up` 输入以下内容：

```
#!/bin/bash
# tinc 在执行脚本的时候会提供 $INTERFACE 变量
# 此处 10.0.1.1/24 是自定义的服务器内网 IP 
# 10.0.1.0/24 是内网的子网掩码
ip address add 10.0.1.1/24 dev $INTERFACE; # 给网络接口一个 IP 地址
ip route add 10.0.1.0/24 dev $INTERFACE;   # 设置路由规则，这个子网下的流量转发到 vpn 去
ip link set $INTERFACE up;     						 # 最后启动这个网络接口
```

编辑 `tinc-down` 输入以下内容

```
#!/bin/bash
ip link set $INTERFACE down; # 关闭网络接口
ip route del 10.0.1.0/24;    # 删除路由规则
```

最后需要对两个文件增加可执行权限：

```
chmod +x tinc-up tinc-down
```

到 `hosts/` 文件夹下，创建本节点的配置文件

```
nano NETNAME
```

在这里可以输入一些可选配置

```
Compression = 9     # 可选压缩等级，具体可看文档
PMTU = 1492         # 可选初始化 MTU 大小
PMTUDiscovery = yes # 可选主动监测 MTU 大小
Port = 1919         # 可选更改监听端口。
```

然后需要生成服务器的公私钥：

```
tincd --net netname -K # -K 后可跟密钥长度
```

到此，服务器的配置完成，在启动 tinc 的 systemd 服务之前，可以先启动起来测试一下看看有没有问题

```
tincd --net netname -d5 -D # 显示调试 log 运行
```

## 客户端配置

需要先把服务器端的 `/etc/tinc/netname/hosts/NETNAME` 复制到客户端的 `/etc/tinc/netname/hosts` 中，而且客户端对应的文件也得复制到服务器的相同位置，因为需要交换公钥，客户端才能登录到 tinc 。

客户端的 `tinc.conf` 配置有一点点不一样：

```
Name = CLIENT_NAME     # 给此节点起个名字
AddressFamily = ipv4   # 使用 ipv4 
Mode = switch          # 使用交换机模式
#Device = /dev/net/tun # 可选，明确使用 tun/tap 设备
ConnectTo = NETNAME    # 连接到目的节点
#Proxy = socks5 127.0.0.1 1082 # 可选的代理手段，见文档
```

`tinc-up` 和 `tinc-down` 也有点不同

```
#!/bin/bash
# tinc-up
ip link set $INTERFACE up
```

```
#!/bin/bash
# tinc-down
ip link set $INTERFACE down
```

还需要针对 `hosts/NETNAME` 作一些修改：

```
Address = 62.210.211.120 # 服务器地址
Port = 1919							 # 端口
Compression = 9     
PMTU = 1492         
PMTUDiscovery = yes 
Port = 1919         
-----BEGIN RSA PUBLIC KEY-----
.............
-----END RSA PUBLIC KEY-----
```

执行 `tincd --net netname -K` 后，把客户端 `hosts/CLIENT_NAME` 复制到服务器端的 `hosts/` 下。然后可以在客户端上启动 tinc 检查一下看看是否能连得上。

## DHCP 配置

现在客户端是没有 IP 的，需要增加 IP 设置才能连通。可以临时给一个子网下的 IP 地址用 ping 命令测试连通性。本文的目标是达到能自动下发 IP 的效果，所以这里直接补充 DHCP 服务器的设置。

回到服务器上，这里使用 `isc-dhcp-server` 作为 DHCP 服务器。

```
sudo apt install isc-dhcp-server
```

安装好后，应该会存在 `/etc/default/isc-dhcp-server` 文件。如果只希望 DHCP 服务器服务于 VPN ，可在其中找到 `INTERFACES` 或者新增然后修改：

```
.... # 其他内容
INTERFACES="netname"
```

如果不在配置中指定，tinc 会在启动的时候使用  tinc 实例的网络名称创建一个虚拟接口，填入对应网络名称即可。

然后修改 dhcp 配置 `/etc/dhcp/dhcpd.conf` :

```
....# 其他配置
# 需要服务的子网及其掩码
subnet 10.0.1.0 netmask 255.255.255.0 {
#	authoritative; # 若 DHCP 服务器为网段内唯一权威服务器，可增加此选项。
  range 10.0.1.100 10.0.1.240;         # DHCP 地址池范围
  option subnet-mask 255.255.255.0;    # 子网掩码
  option routers 10.0.1.1              # 服务器内网 IP
# option domain-name-servers 10.0.1.1; # 如果有内网 DNS 服务，可选。  
}
```

然后重启 dhcp 服务

```
systemctl restart isc-dhcp-server
```

dhcp 服务在没有 interface 或者 interface 关掉之后会自己直接退出，可以在 `tinc-up` 和 `tinc-down` 中增加对服务的控制：

```
systemctl start isc-dhcp-server # 增加在服务器 tinc-up 的最后
systemctl stop isc-dhcp-server  # 增加在服务器 tinc-down 的第一句命令之前
```

客户端的 dhcp 服务需要监听 vpn 所在的 interface 的情况及时作出动作，同样可以在 `tinc-up` 和 `tinc-down`  脚本内增加。在有网络管理器的系统上可能因为 dhcp 客户端与网络管理器之前的互相依赖关系不便直接在脚本内通知，那可以修改一下两个脚本的内容。

### 配合一下 NetworkManager 的管理

在使用 NetworkManager 的 ubuntu 上可以修改一下 `tinc-up`

```
# tinc-up:
ip link set $INTERFACE up
nmcli device modify netname ipv4.method auto # 增加这句
```

如果是 mac ，可以使用 `ipconfig set tap0 DHCP` （ OS X 上 tinc 会使用 tap0 作为 interface）

此时应该就可以收到服务器下发的地址且能连通服务器了。

## 配置桥来共享内网

在客户端，通过配置桥来共享 tinc vpn 内网是个好玩同时也挺实用的做法，例如：电脑上有虚拟机，这样可以把虚拟机接进桥然后与在其他地区的节点通讯；有多余的网口，与无线 AP 或者交换机连接，连接上 AP 或者交换机的设备能直接获得 IP 并访问内网。

不同的系统有不同的配置方式，这里我以 ubuntu 18.04 为例，假设客户端电脑上有个多余的网口 `eth1` 用来接 AP。

桥（bridge）的配置需要使用 `brctl` 命令操作，如果没有则需要安装：

```
apt install bridge-utils
```

然后可以在客户端新建一个桥

```
brctl addbr br0 # 这里就叫它 br0，可以随便，后面命令桥名一致即可
```

桥就是一个交换机，默认有一个口，就是桥所在的主机自己的口。而往桥添加更多的接口就是给交换机增加更多的口。

新建完之后，需要修改一下 `tinc-up` 和 `tinc-down` 的设置

```
# tinc-up:
ip link set $INTERFACE up && \
ip addr replace 0.0.0.0 dev $INTERFACE && \ # 清除掉 VPN interface 的 IP 设置。
brctl addif br0 $INTERFACE eth1 && \        # 把 VPN interface 和 eth1 加入到桥
ip link set br0 up  # 启动桥
```

```
# tinc-down:
brctl delif $INTERFACE eth1 && \ # 移除在桥上的接口
ip link set br0 down && \        # 关闭桥
ip link set $INTERFACE down      
```

客户端对应的 dhcp 需要把服务的 interface 从 vpn interface 改成桥。主机从桥访问 vpn 内网，AP / 交换机的设备从 eth1 通过 br0 桥接到 vpn interface 。

在句尾加 ` && \` 是为了让全部命令成为一句连贯的话并顺序执行，中途有一句失败了可以直接停止执行，可选。

### 配合 NetworkManager 的管理

dhcp client 直接改 interface 为 br0 是可以的，还有另外一种方法，就是让 NetworkManager 托管桥。

用 `nmcli` 创建一座桥，同时添加 `eth1` 到桥上，管理配置命名为 br0 ，桥命名为 br0 ：

```
nmcli conn add type ethernet slave-type bridge con-name br0 ifname eth1 master br0
```

使用 nmcli 管理桥之后，就不方便随意用 `brctl` 去启动或者停止桥，所以对桥的大部分操作都需要换用 `nmcli`

```
# tinc-up:
ip link set $INTERFACE up && \
ip addr replace 0.0.0.0 dev $INTERFACE && \
nmcli conn up br0 && \     # 启动桥
brctl addif br0 $INTERFACE # 单独添加 vpn interface 到桥，tinc 自己管理此 interface 
```

```
# tinc-down:
ip link set $INTERFACE down && \
brctl delif br0 $INTERFACE && \ # 从桥上拔掉 vpn interface（其实概念跟拔 WAN 口很像哦 wwww）
nmcli conn down br0 # 关闭网桥
```

可以用 `nmcli conn show --active` 看看当前网络连接的情况。

### 可能出现的问题

有可能运行起来后，桥上的主机可以正常分配 IP 而其他在桥上的设备不行。根据 Google 来的信息，看见有个 mail-list ([1](https://www.tinc-vpn.org/pipermail/tinc/2018-August/005280.html),[2](https://www.tinc-vpn.org/pipermail/tinc/2018-August/005281.html)) 里提到说有可能是因为内核的设置导致桥部分包的转发被拦截。继续根据关键设置项的名称查找，找到 [libvirt](https://wiki.libvirt.org/page/Net.bridge.bridge-nf-call_and_sysctl.conf) 所的信息：

```
Most virtual guests configured using libvirt connect to the network via a Linux host bridge; the behavior of the bridge changes depending on the setting of 3 "tunables" recognized by the kernel bridge module:

- net.bridge.bridge-nf-call-arptables
- net.bridge.bridge-nf-call-ip6tables
- net.bridge.bridge-nf-call-iptables

These control whether or not packets traversing the bridge are sent to iptables for processing. In the case of using bridges to connect virtual machines to the network, generally such processing is *not* desired, as it results in guest traffic being blocked due to host iptables rules that only account for the host itself, and not for the guests.

However, the bridge module in the kernel has the default for all three of these values set to "1" ("on", i.e. "*do* send the packets to iptables"), and for historical reasons the kernel maintainers refuse to change this default (see http://patchwork.ozlabs.org/patch/29319/ )

After the rejection of the above change to the compiled-in defaults by the kernel, many Linux distros (including Fedora, RHEL, and CentOS) made an attempt to solve this problem by adding lines to /etc/sysctl.conf to modify the default settings compiled into the bridge module:

  net.bridge.bridge-nf-call-arptables = 0
  net.bridge.bridge-nf-call-ip6tables = 0
  net.bridge.bridge-nf-call-iptables = 0

```

可以直接修改 `sysctl.conf` ，增加上述三个配置后应用。我的做法是在 `tinc-up` 上增加三句话：

```
# tinc-up
echo 0 > /proc/sys/net/bridge/bridge-nf-call-arptables
echo 0 > /proc/sys/net/bridge/bridge-nf-call-ip6tables
echo 0 > /proc/sys/net/bridge/bridge-nf-call-iptables

...
```

需要同时检查服务器和客户端上这三个选项是否已经设为关闭。设置完成后如果没看到效果，重启一下各个服务就好了。

## 其他玩法？

例如在里面塞个域名解释服务专门作内网域名解释以及躲避 DNS 污染、增加 NAT 转发功能实现公网访问等等。tinc 还能通过组合桥接多个网络，可以自己做一个大内网邀请大家来玩 wwww（例如互相加内网互相搭游戏服务器在内网对战、组成工作网路或者视频语音网络等等等等……）
